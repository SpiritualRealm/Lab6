### **🔹 Step 1: Understanding IPTables Chains**

#### **Questions**: 
### i. IPTables Built-in Chains and Their Purposes

IPTables has three built-in chains:

- **INPUT**  
  Handles incoming packets that are destined for the local machine itself.  
  Example: traffic coming from the Attack VM to the Defense VM.

- **FORWARD**  
  Handles packets that are passing through the machine to another destination.  
  This is mainly used when the machine acts as a router or gateway.

- **OUTPUT**  
  Handles packets generated by the local machine and sent outward.  
  Example: traffic originating from the Defense VM going to another VM.

---

### ii. Difference Between DROP and REJECT

- **DROP**  
  Silently discards the packet without sending any response back to the sender.  
  The sender receives no indication of whether the packet was blocked.

- **REJECT**  
  Blocks the packet but also sends an error response back to the sender, such as:
  - ICMP destination unreachable
  - TCP reset (RST)

This lets the sender know the traffic was denied.

---

### iii. When Each Policy Is Typically Used

- **DROP** is typically used when you want to hide the presence of the machine or service from attackers.
  This is common in security-focused firewall rules.

- **REJECT** is typically used when you want legitimate users or systems to know the connection was denied immediately.
  This is often helpful in troubleshooting and controlled network environments.

## How DROP Differs from REJECT

### REJECT
The firewall blocks the packet and sends an ICMP error response back to the sender.

Example:

`Destination unreachable (Port unreachable)`

This informs the sender immediately that the packet was denied.

---

### DROP
The firewall silently discards the packet.

No response is sent back.

The sender continues waiting until the request times out.

This makes it appear as though the destination host is not responding.

---

## Why Choose One Over the Other?

### Use DROP
A firewall administrator may use DROP for better security because it hides the existence of the host or service.

This is common in hardened environments.

---

### Use REJECT
A firewall administrator may use REJECT when they want users or systems to know immediately that access is denied.

This is useful for troubleshooting and internal networks.

---

## Behavior Summary

- **REJECT:** blocks and sends an error response
- **DROP:** blocks silently with no response

This step clearly demonstrates the functional difference between the two firewall actions.

### **🔹 Step 1: Configuring ICMP Filtering**  

#### **Questions**:
## Scenario 1: Attack VM Pings Defense VM

In this scenario, the ping from the Attack VM to the Defense VM was successful.

The Wireshark capture shows alternating:

- `Echo (ping) request`
- `Echo (ping) reply`

This confirms that the following rules are working correctly:

```bash
sudo iptables -A INPUT -p icmp --icmp-type echo-request -j ACCEPT
sudo iptables -A OUTPUT -p icmp --icmp-type echo-reply -j ACCEPT
```

The Defense VM is allowed to receive ping requests and send ping replies.

---

## Scenario 2: Defense VM Pings Attack VM

In this scenario, no packets appeared in Wireshark.

This is expected behavior.

When the Defense VM tries to ping the Attack VM, it generates an **ICMP echo request**.

However, the OUTPUT chain only allows:

`icmp echo-reply`

and then drops all other ICMP traffic:

```bash
sudo iptables -A OUTPUT -p icmp -j DROP
```

Because of this, the outgoing echo request is blocked before reaching the network interface.

As a result, Wireshark shows no packets.

This confirms that the OUTPUT DROP rule is functioning correctly.

---

## How Filtering Impacts ICMP Communication

ICMP filtering controls which types of ICMP packets are allowed to enter or leave the machine.

In this configuration:

- incoming echo requests are allowed
- outgoing echo replies are allowed
- all other ICMP traffic is dropped

This allows other hosts to ping the Defense VM while preventing the Defense VM from initiating its own pings.

---

## Importance of Rule Order

Rule order is extremely important in IPTables because rules are processed **top to bottom**.

The firewall checks each packet against the first matching rule.

For example:

```bash
ACCEPT icmp echo-request
DROP   icmp
```

The specific ACCEPT rule must come first.

If the DROP rule came first, it would block all ICMP traffic before the echo-request rule could be matched.

This is why more specific rules should always be placed before broader rules.

### **🔹 Step 3: Turning Defense VM into a Web Server** and **Step 4: Securing HTTP Traffic with IPTables**  


#### **Questions**:
## How `-m state --state NEW,ESTABLISHED` Works

The `-m state` module allows IPTables to filter packets based on the connection state.

### NEW
`NEW` means the packet is attempting to start a new connection.

For TCP, this is typically the initial **SYN** packet of the three-way handshake.

Example:

```text
[SYN]
```

---

### ESTABLISHED
`ESTABLISHED` means the connection has already been created and the packet belongs to an active session.

Examples include:

- SYN/ACK
- ACK
- HTTP GET / response packets

This is crucial for TCP because once the connection begins, all subsequent packets must be allowed for communication to continue.

---

## Why INPUT Uses `NEW,ESTABLISHED` but OUTPUT Uses Only `ESTABLISHED`

### INPUT Rule (port 8000)
```bash
--state NEW,ESTABLISHED
```

This rule must allow:

- **NEW** → incoming client connection requests
- **ESTABLISHED** → later packets in the session

Without `NEW`, the initial SYN packet would be blocked.

---

### OUTPUT Rule
```bash
--state ESTABLISHED
```

The server is not initiating the connection.

It is only responding to an already established session.

Therefore only `ESTABLISHED` traffic is required.

This includes:

- SYN/ACK
- ACK
- HTTP response packets

---

## Key Differences: Port 8000 vs Port 8001

### Port 8000 (Allowed)

Wireshark shows a full TCP session:

- SYN
- SYN, ACK
- ACK
- HTTP GET
- HTTP 304 / response
- FIN, ACK

This confirms that the connection was successfully established and HTTP traffic was allowed.

---

### Port 8001 (Blocked)

Wireshark shows:

- SYN
- repeated SYN retransmissions

No SYN/ACK or HTTP response appears.

This means the firewall dropped the traffic.

The connection could not be established.

---

## Three-Way Handshake Impact

### Allowed Connection (8000)

The complete TCP handshake occurs:

```text
SYN → SYN/ACK → ACK
```

After this, HTTP communication proceeds normally.

---

### Blocked Connection (8001)

The handshake fails at the first step:

```text
SYN → no response
```

The client retransmits SYN packets until timeout.

This demonstrates how the firewall directly controls whether TCP sessions can be established.
